## 测试用例 1: 对话框禁用输入功能

### 测试目标
验证任务执行期间输入框的禁用/恢复机制是否正常工作

### 前置条件
- 用户已登录
- 已选择工作目录
- 已创建或选择会话

### 测试步骤

#### 1.1 任务开始时输入框禁用
**步骤**:
1. 打开聊天页面
2. 在输入框输入测试消息
3. 点击发送按钮,启动任务
4. 立即检查输入框状态

**预期结果**:
- 输入框应该被禁用 (`disabled` 属性为 true)
- 输入框显示占位符文本: "创建/选择任务后开始..."
- 发送按钮变为停止按钮(红色)
- 用户无法在输入框中输入文字

**验证点**:
- `textarea[data-prompt-input]` 元素的 `disabled` 属性
- 按钮图标从发送箭头变为停止方块
- 按钮背景色从 accent 变为 error

#### 1.2 任务结束时输入框恢复
**步骤**:
1. 等待任务执行完成
2. 检查输入框状态

**预期结果**:
- 输入框恢复可用状态 (`disabled` 属性为 false)
- 输入框显示正常占位符
- 发送按钮恢复为发送状态(蓝色)
- 用户可以正常输入文字

**验证点**:
- `textarea[data-prompt-input]` 元素的 `disabled` 属性为 false
- 按钮图标恢复为发送箭头
- 按钮背景色恢复为 accent

#### 1.3 边界条件测试

**测试场景 A: 任务中断**
1. 启动任务
2. 点击停止按钮中断任务
3. 检查输入框状态

**预期**: 输入框应该恢复可用

**测试场景 B: 任务失败**
1. 启动一个会失败的任务(如访问不存在的文件)
2. 等待任务失败
3. 检查输入框状态

**预期**: 输入框应该恢复可用

**测试场景 C: 快速连续发送**
1. 发送第一条消息
2. 在任务执行期间尝试发送第二条消息
3. 验证第二条消息被阻止

**预期**: 输入框禁用,无法发送第二条消息

### 测试数据
- 测试消息: "列出当前目录的文件"
- 失败任务消息: "读取 /nonexistent/file.txt"

### 代码位置
- 组件: `src/ui/components/PromptInput.tsx`
- 关键代码: 第 508 行 `disabled={disabled && !isRunning}`

## 测试用例 2: 文件 hover 按钮功能

### 测试目标
验证文件浏览器中 hover 到文件时显示的操作按钮功能

### 前置条件
- 用户已登录
- 已选择工作目录
- 文件浏览器已打开并显示文件列表

### 测试步骤

#### 2.1 Hover 显示按钮
**步骤**:
1. 打开文件浏览器
2. 将鼠标悬停在文件项上
3. 观察是否显示操作按钮

**预期结果**:
- 显示"打开"按钮
- 显示"复制文件名"按钮
- 按钮位置合理,不遮挡文件名
- 按钮有 hover 效果

**验证点**:
- 按钮元素可见性
- 按钮文本或图标正确
- CSS 动画效果流畅

#### 2.2 "打开"按钮功能
**步骤**:
1. Hover 到文件上
2. 点击"打开"按钮
3. 验证文件是否在编辑器中打开

**预期结果**:
- 文件在系统默认编辑器中打开
- 或在应用内编辑器中打开(如果有)
- 无错误提示

**验证点**:
- IPC 调用 `workspace.openFile` 成功
- 文件路径正确传递

#### 2.3 "复制文件名"按钮功能
**步骤**:
1. Hover 到文件上
2. 点击"复制文件名"按钮
3. 粘贴到输入框验证

**预期结果**:
- 文件名(或完整路径)被复制到剪贴板
- 显示成功提示(toast 或其他)
- 粘贴内容与文件名一致

**验证点**:
- 剪贴板内容正确
- 成功提示显示
- 复制的是文件名还是完整路径(需确认需求)

#### 2.4 交互细节测试

**测试场景 A: 快速 hover**
1. 快速移动鼠标经过多个文件
2. 验证按钮显示/隐藏流畅

**预期**: 无闪烁,动画流畅

**测试场景 B: 目录 vs 文件**
1. Hover 到目录上
2. Hover 到文件上
3. 对比按钮显示

**预期**: 目录和文件可能显示不同的按钮(需确认需求)

**测试场景 C: 按钮位置**
1. Hover 到长文件名的文件
2. Hover 到短文件名的文件
3. 验证按钮位置一致性

**预期**: 按钮位置固定,不随文件名长度变化

### 测试数据
- 测试文件: `package.json`, `README.md`, `src/ui/App.tsx`
- 长文件名: `very-long-filename-for-testing-button-position.ts`

### 代码位置
- 组件: `src/ui/components/workspace/FileExplorer.tsx`
- 状态: **未实现** (需在任务 #2 中实现)

## 测试用例 3: 模型介绍功能

### 测试目标
验证模型介绍信息在桌面端正确显示,后台可以设置,数据库正确保存

### 前置条件
- 用户已登录
- 后台管理系统可访问
- 数据库连接正常

### 测试步骤

#### 3.1 桌面端显示模型介绍
**步骤**:
1. 打开模型选择器
2. 查看模型列表
3. 验证是否显示模型介绍(note 字段)

**预期结果**:
- 每个模型下方显示介绍文字
- 介绍文字样式清晰可读
- 介绍文字不遮挡其他信息

**验证点**:
- `note` 字段内容显示在 UI 中
- 文字样式: `text-xs text-muted`
- 位置: 模型名称和价格信息之间或之下

#### 3.2 后台管理系统设置模型介绍
**步骤**:
1. 登录后台管理系统
2. 进入模型管理页面
3. 编辑某个模型的介绍信息
4. 保存修改

**预期结果**:
- 可以编辑模型介绍字段
- 保存成功提示
- 修改立即生效

**验证点**:
- API 调用成功 (PUT /api/admin/models/:id)
- 返回状态码 200
- 响应包含更新后的数据

#### 3.3 数据库正确保存
**步骤**:
1. 在后台设置模型介绍
2. 查询数据库验证
3. 刷新桌面端验证

**预期结果**:
- 数据库 `models` 表的 `note` 字段更新
- 桌面端刷新后显示新的介绍
- 数据一致性保持

**验证点**:
- 数据库查询: `SELECT note FROM models WHERE id = ?`
- API 响应: `/api/models` 返回的 `note` 字段
- 前端缓存刷新机制

#### 3.4 边界条件测试

**测试场景 A: 空介绍**
1. 设置模型介绍为空字符串
2. 验证桌面端显示

**预期**: 不显示介绍区域,或显示默认文本

**测试场景 B: 长介绍**
1. 设置超长介绍文字(500+ 字符)
2. 验证桌面端显示

**预期**: 文字截断或滚动,不破坏布局

**测试场景 C: 特殊字符**
1. 设置包含 HTML/Markdown 的介绍
2. 验证是否正确转义

**预期**: 特殊字符被转义,不执行脚本

### 测试数据
```json
{
  "modelId": "claude-opus-4-6",
  "note": "最强大的模型,适合复杂推理任务。上下文窗口 200K tokens。"
}
```

### 代码位置
- 前端组件: `src/ui/components/ModelSelector.tsx`

## 附录: 测试工具和命令

### Playwright 命令
```bash
# 安装依赖
npm install -D @playwright/test

# 安装浏览器
npx playwright install

# 运行所有测试
npx playwright test

# 运行特定测试
npx playwright test tests/e2e/input-disabled.spec.ts

# 调试模式
npx playwright test --debug

# 生成测试代码
npx playwright codegen http://localhost:3000

# 查看报告
npx playwright show-report
```
